一、六大建设域

构建多 Agent 协作体系的六大核心领域

域一：编排架构层（Orchestration Fabric）

核心决策：选择什么架构范式？

范式适用场景推荐度
Hierarchical（分层编排）合规审计、确定性流程为主★★★★★
Hybrid（混合架构）复杂业务，部分确定+部分灵活★★★★★
Peer-to-Peer（分布式自治）研究型、开放域问题★★★
推荐：业务绑定场景必选 Hybrid 架构
  • 顶层：集中式 Orchestrator（负责任务分解、路由决策、全局状态）
  • 中层：按业务域划分 Sub-Agent Group（如"客服域"、"风控域"）
  • 底层：每个 Agent 内部可自治

推荐技术栈

  • 编排框架：LangGraph v1.0（状态机+checkpoint最成熟）或 Microsoft Agent Framework（Azure生态）
  • 跨框架互操作：Google A2A 协议（JSON-RPC 2.0）
  • 工具连接：MCP（Model Context Protocol，250+ 标准工具）

各层建设要点

层次建设内容推荐技术
L1 Prompt/Fine-tuningAgent 角色定义、系统提示模板、LoRA 微调7B学生模型+72B教师蒸馏
L2 RAGSOP/产品文档/政策文档 → 向量化 → 检索注入LlamaIndex + Qdrant
L3 知识图谱实体关系建模（客户↔订单↔规则↔合规）→ 多跳推理Neo4j + GraphRAG
L4 规则引擎硬规则（价格上限、合规限制）→ OPA/Cedar 确定性执行OPA + 自然语言→POJO 转换
关键原则：
- LLM 只用于意图理解和推理，不用于运行时业务决策
- 高业务风险操作必须由确定性代码执行（Drools/OPA）
- Agent 决策结果写回图谱形成记忆闭环

域三：记忆与状态层（Memory & State）

三层记忆架构：

全局共享记忆 (Shared Knowledge)

领域知识图谱 + 政策文档 + 产品目录

Neo4jVector DBTTL: 月级/年级

长期情景记忆 (Episodic Memory)

跨运行的历史: 谁、何时、做了什么 | 用户偏好 + 失败尝试 + 人工审批记录

pgvectorWeaviate隐私隔离

短期工作记忆 (Scratch Memory)

单次运行内的中间工具结果 + 部分计划

Redis内存TTL: 任务结束销毁
状态管理铁律：
  • Agent 必须无状态，每次从记忆重构上下文
  • 共享状态使用 Reducer 模式解决并行写冲突
  • Checkpoint 持久化：生产用 Postgres，不用 SQLite（并发损坏）
  • 状态精简：仅写入不可重建的数据

域四：通信与工具层（Communication & Tools）

Agent 间通信协议层次

跨系统 A2A Protocol (JSON-RPC + SSE + gRPC)
框架内 StateGraph/LangGraph 共享状态
事件驱动 Redis Streams / Kafka (Pub-Sub)
直接调用 函数调用 / RPC (低延迟场景)

通信安全规则

  • Agent 间传输必须使用结构化 Schema（JSON Schema 验证），禁止自由文本交接
  • 每次交接带完整上下文和版本号
  • W3C Trace Context 传播（跨 Agent 追踪 trace_id/span_id）
  • JWT/HMAC 签名验证 + 授权白名单
  • 跳数计数器（防循环委派 A→B→C→A）

工具注册发现层次

硬编码 框架内工具
标准化 MCP Server
动态 Agent 注册表（Service Mesh + A2A AgentCard）

域五：安全治理层（Security & Governance）

五层防御体系（Defense-in-Depth）：

Layer 0 Design-Time - Agent 权限边界、工具白名单定义
Layer 1 Input Guardrails - Prompt注入检测 + PII脱敏
Layer 2 Action Plan Guardrails - 意图校验 (OPA策略)
Layer 3 Output Guardrails - Schema验证 + 合规检查
Layer 4 Human-in-the-Loop - 高风险审批 + 四眼原则
Layer 5 Audit + Eval - 不可变审计日志 + 回归测试

安全域详情

安全域必须做工具推荐
Prompt 安全输入过滤、注入检测、签名化系统提示Lakera Guard / NeMo Guardrails
工具安全最小权限、短凭证（≤15min）、沙箱隔离MCP Server + 策略引擎
记忆安全来源标记、完整性校验、租户隔离、追加审计PostgreSQL + 权限控制
通信安全JWT签名、信任级别(4级)、防重放(>5min拒绝)OAuth 2.0 + W3C Trace
Agent 安全Step预算、计划偏离检测、熔断器(5次→60s)LangGraph内置
合规审计不可变日志(5-7年)、实时策略匹配、越权升级ValidMind / 自建

总结

建设复杂多Agent体系的核心要义

建设复杂多 Agent 体系的本质是 将业务流程确定性化，将灵活推理局部化。

1 业务规则与 Agent 推理的边界隔离
硬规则用 OPA/Drools，灵活推理用 LLM，绝不混用
2 状态与记忆的三层分离
短期/长期/共享，访问模式截然不同
3 全链路可观测 + 评估闭环
没有评估的 Agent 系统是黑箱，不可审计等于不可控
